# TripIn Web 首页帖子流重构设计

## 背景

当前 Web 首页存在明显问题：

- 首屏依赖大面积 hero 和超大标题，内容到达太慢
- 左右卡片尺寸不一致，容易形成难看的留白
- 页面更像“产品介绍页”或“后台前台混合页”，不是可连续浏览的社区首页
- 帖子没有形成完整阅读单元，首页必须依赖“点进详情”补足体验
- 视觉语言偏纸张感，厚重、发黄、信息密度失衡，不符合用户希望的简约白色路线社区风格

本次重构只聚焦 **Web 首页帖子流**。路线编辑页后续单独重构。

## 目标

把 Web 首页重构为：

- `单列居中连续刷帖`
- 首页本身就是主要消费场景，而不是详情页导流页
- 每条帖子都是一个完整可操作的路线阅读器
- 页面结构尽量接近未来移动端，只区分鼠标交互和页面切换方式

首页应当让用户一打开就进入内容流，连续浏览路线、切换点位、切换图片，而不是先读大段说明。

## 非目标

本次不做这些事情：

- 不重构路线编辑页
- 不重构后端接口结构
- 不引入新的真实地图供应商能力
- 不新增登录、推荐算法、发布流程
- 不在这一轮做蓝绿渐变主视觉

说明：

- 产品的长期特别色调仍然可以保留 `主蓝 + 次绿` 的方向
- 但这一版首页先以 `简约白色基底 + 轻地图感` 为主，不强推蓝绿渐变

## 首页结构

首页整体采用：

- `极薄顶部`
- `单列居中帖子流`
- `很弱的详情入口`

### 1. 极薄顶部

顶部只保留非常轻的品牌存在感，不做 hero，不做说明卡，不做传统门户导航。

可包含：

- `TripIn` 字标
- 当前页面标识
- 低存在感的个人入口或搜索入口

限制：

- 不允许占据首屏主要视觉面积
- 不允许以标题、口号、产品说明抢走内容流入口

### 2. 单列居中帖子流

首页主体是连续刷帖的内容流：

- 采用单列居中布局
- 不做瀑布流
- 不做左右不等高双栏
- 不做首屏产品介绍后再进入帖子流

节奏要求：

- 第一条帖子更强，视觉权重更高
- 后续帖子结构一致，但高度和留白更克制
- 所有帖子都必须保持“完整可操作”，不能把后续帖子退化成摘要卡

### 3. 很弱的详情入口

首页已经能完成主要浏览和切换，因此详情页入口必须很弱：

- 可以保留
- 但不应成为首页的主 CTA
- 不能迫使用户“必须点进去才能看完整内容”

## 单条帖子结构

每条帖子是一个完整内容单元，包含四个区域。

### 1. 顶部作者区

布局参考 Instagram：

- 左侧圆形头像
- 右侧作者名
- 作者名下方一行小字：`TripIn · 城市名`

要求：

- 区域很轻
- 不抢主体内容
- 作用是快速建立作者与地点上下文

### 2. 路线导航区

帖子上半部分由两块组成：

- 左侧 `路线图`
- 右侧 `点列表`

比例目标：

- `路线图 : 点列表 = 3 : 1`

约束：

- 两块高度严格一致
- 两块必须组成一个整体，不允许视觉断裂

#### 路线图职责

路线图不是导航页，也不是后台地图控件，而是内容阅读器中的路线导航层。

需要支持：

- 展示整条路线
- 展示所有点
- 当前点高亮
- 左右箭头切点
- 点击点切换当前点

视觉要求：

- 白色基底上的轻地图感
- 有一点空间感 / 地球仪感 / 轨迹感
- 不做沉重底图，不做导航软件风格

#### 点列表职责

点列表是路线图的同步信息面板：

- 展示点标题
- 展示地点信息
- 展示时间信息
- 点击列表项切换当前点
- 自动跟随当前点滚动
- 允许用户手动滚动

列表窗口默认可容纳约 `4` 个点，但布局必须按点数自适应：

- `1 个点`：单个点填满整个点列表区
- `2 个点`：对称填充整个列表区
- `3 个点`：三段式均衡填充
- `4 个及以上`：固定窗口高度，内部滚动

原则：

- 不能因为点少出现大片空洞
- 不能因为点多把整条帖子撑炸

### 3. 当前点图片区

图片区位于路线导航区下方，是帖子最强视觉区。

默认规则：

- 默认显示 `当前点的第一张图片`
- 切换当前点时，图片自动切回该点默认图
- 同一点下允许左右切换多张图片

图片区不是普通轮播图，而是被当前点状态驱动的图像展示区。

### 4. 很弱的底部操作区

第一版保留弱操作位即可：

- 进入详情
- 点赞
- 收藏

要求：

- 不抢内容
- 不做强工具栏

## 交互规则

### 1. 当前点是帖子内部的主状态

每条帖子维护一个 `当前点`，驱动三处同步：

- 路线图当前高亮点
- 点列表当前高亮项
- 当前点对应的图片组

要求：

- 三者必须由同一个状态驱动
- 不允许路线图、列表、图片区各自独立

### 2. 切点入口

当前点可以通过三种方式切换：

- 点击路线图上的点
- 点击点列表中的点
- 点击路线图左右箭头

切换后必须同步：

- 高亮点变化
- 点列表滚到可见区域
- 图片切换到该点默认图

### 3. 点切换动画

当前点切换不能硬切。

建议动画表现：

- 点位从非选中到选中时微放大
- 选中点外层有柔和扩散感
- 连线局部强调
- 图片以短促平滑过渡切换

原则：

- 动画要表现“状态流动”
- 不要做厚重的 dashboard 动效

### 4. 图片切换规则

当前点下的图片可左右切换。

箭头规则：

- 当图片还没到该点的边界时，箭头执行“同点内切图”
- 当图片已经到该点最后一张时，右箭头进入“可切到下一个点”的提示状态
- 当图片已经到该点第一张时，左箭头进入“可切到上一个点”的提示状态
- 如果当前点已经是整条路线的边界点，且图片也在边界，箭头直接消失

目标：

- 箭头既承担图片切换，也承担跨点切换提示

### 5. 帖子状态隔离

首页上每条帖子都必须是独立阅读器：

- A 帖子的当前点不影响 B 帖子
- A 帖子的当前图片索引不影响 B 帖子

首页最终是：

- `多个独立路线阅读器的纵向排列`

## 视觉方向

### 1. 总体视觉

整体方向定义为：

- `简约白色基底`
- `轻地图气质`
- `Instagram 式内容节奏`

### 2. 必须避免的问题

这轮重构明确禁止以下反模式：

- 超大 hero
- 夸张标题
- 左右尺寸失衡的大卡拼接
- 因块状不等高导致的大面积难看留白
- 纸张感过强的暖黄背景
- 像后台或 dashboard 一样的组件堆叠

### 3. 色彩

第一版采用：

- 主背景：干净偏暖白
- 辅助层：浅灰白
- 线框：极细浅边框
- 阴影：非常克制的柔和阴影

蓝绿作为未来品牌特色可保留，但这一版不强制上蓝绿渐变主视觉。

### 4. 字体与标题

要求：

- 标题尺寸必须明显收敛
- 首页不依赖大字形成视觉压迫
- 视觉中心应该来自：
  - 当前点图片
  - 路线图状态变化
  - 点列表节奏

## 首页数据与演示要求

为了验证布局适应性，首页第一版不能只展示 1 条帖子。

至少应准备多条演示帖子，覆盖：

- `1 个点`
- `2 个点`
- `3 个点`
- `4 个点`
- `6 个点以上`

同时建议覆盖不同图片数量组合：

- 单点单图
- 单点多图
- 多点单图
- 多点多图

目的：

- 验证点列表填充策略
- 验证图片切换逻辑
- 验证连续刷帖的节奏是否成立

## 跨端复用原则

这次 Web 首页重构必须为移动端提供可复用结构，而不是只做桌面特化页面。

移动端后续应尽量复用：

- 帖子信息结构
- 当前点驱动模型
- 路线图 / 点列表 / 图片区的同步关系
- 点数量自适应逻辑
- 图片与点切换规则

Web 与移动端的主要区别应只在：

- 输入方式：鼠标 vs 手势
- 页面切换方式
- 局部尺寸和密度

而不是重新发明一套完全不同的帖子结构。

## 实现边界

本轮实现只做：

- Web 首页结构重构
- 首页帖子组件重构
- 首页演示数据扩充
- 首页帖子内部交互重构

本轮不做：

- 编辑页重构
- 发布页重构
- 后端协议改造
- 推荐系统逻辑升级

## 验收标准

首页重构完成后，至少满足以下标准：

- 一打开页面直接进入帖子流
- 不再存在大面积 hero 和大标题压屏
- 每条帖子在首页内部即可完成切点、切图和基础浏览
- 点少时不空，点多时不乱
- 首条和后续帖子的节奏层次清楚
- Web 视觉足够简洁，不土，不像后台，也不像纸张风 landing page
- 页面结构可以自然映射到移动端
