## feature/mq-prefs-push-search — message queue + follow + notifications + preferences + search (2026-06-15)
- Backend: added a Redis Streams message queue. New `config/RedisConfig.java` (parses `REDIS_URL`, default `redis://localhost:6379`), `mq/` package — `DomainEvent` (post_liked/post_commented/user_followed), `EventPublisher` (XADD to stream `tripin:events`, **auto-degrades to synchronous handling when Redis is unavailable** so notifications never drop), `MqConsumer` (consumer group `tripin-workers`, pings Redis at startup and only starts the listener if reachable), `MqStatus` (shared availability flag). Added `spring-boot-starter-data-redis` to `pom.xml`.
- Backend: in-app notification center. New `Notification` table + `NotificationService` (consumer side turns events into notifications, respecting the recipient's preferences and skipping self-actions) + `NotificationsController` (`GET /v1/notifications`, `/unread-count`, `POST /{id}/read`, `/read-all`). Wired `EventPublisher` into `InteractionsService` (publishes on like — only when newly inserted — and on comment).
- Backend: follow feature. New `UserFollow` table + `FollowService` (follow/unfollow, follower/following counts, status; publishes `user_followed` on a new follow) + `GET/POST/DELETE /v1/users/{id}/follow`.
- Backend: user preferences. New `UserPreference` table + `PreferenceService` (notify toggles for likes/comments/follows, `feedScope` all/following, language) + `GET/PUT /v1/users/me/preferences`. The notification consumer consults these before creating a notification.
- Backend: unified search. `SearchService` + `GET /v1/search?q=&type=all|posts|users|places` — posts (title/summary/city via ILIKE, ACTIVE+PUBLIC only), users (username/displayName via ILIKE), places (reuses AMap `PlacesService`).
- DB: `schema.sql` extended with `UserFollow` / `Notification` / `UserPreference` (all `CREATE TABLE IF NOT EXISTS` + FK `DO` block + indexes; additive and idempotent). Applied to the live server DB.
- Web: new pages `/search`, `/notifications`, `/me/preferences`; new components `notifications/NotificationBell` (unread badge) + `NotificationsScreen`, `search/SearchScreen` (debounced, tabbed posts/users/places), `profile/PreferencesScreen` (toggle switches + feed-scope segment), `profile/FollowButton`. Home top bar gained a search icon + notification bell (when logged in); user profile pages gained follower/following counts + a follow button; `/me` now shows 粉丝/关注 counts and a 设置 link. New API client methods + types; new CSS for all of the above (metallic-blue switches, search/notification/preference layouts).
- Server: `REDIS_URL=redis://redis:6379` added to the api service in the server compose; api container recreated with the new jar. Verified end-to-end on the live server: follow → Redis Streams event → consumer → notification (async), plus preferences read/write, follow counts, and search (incl. AMap places). MQ consumer logs `MQ consumer started`. Frontend smoke-tested via preview (no console errors).
